# Full-Adder-Probabilistic-Model

This repository studies a 16-bit ripple-carry adder from both a digital logic and a probabilistic modeling perspective. The project combines:

- a SystemVerilog implementation of a 16-bit full adder,
- a SystemVerilog testbench that drives 10,000 random input samples,
- a Python vector generator,
- and a Python analysis script that parses the VCD dump, compares empirical behavior against theory, and generates plots and tables.

The main goal is to show that the simulated adder not only works functionally, but also matches the expected statistical behavior of a ripple-carry adder under random inputs.

## Repository Overview

### Core source files

- `16bit_fa.sv`
  - SystemVerilog implementation of a 1-bit full adder and a 16-bit ripple-carry adder named `adder16`.
  - Uses only combinational logic.
  - Produces `Sum[15:0]` and `Cout`.

- `tb.sv`
  - SystemVerilog testbench for `adder16`.
  - Reads random input vectors from `test_vectors.mem`.
  - Applies 10,000 samples to the DUT.
  - Dumps waveform activity to `adder16_random.vcd`.
  - Logs sample-by-sample results to `simulation_results.csv`.
  - Checks each sample against the expected arithmetic result and reports mismatches.

- `test_vector_gen.py`
  - Generates reproducible random 16-bit input vectors.
  - Produces:
    - random `A`,
    - random `B`,
    - random `Cin`.
  - Writes these inputs to `test_vectors.mem`.
  - Uses a fixed default seed so results are reproducible.

- `model_scripy.py`
  - Main probabilistic analysis and reporting script.
  - Derives the theoretical model for carry propagation and sum behavior.
  - Parses the VCD file produced by the testbench.
  - Reconstructs internal carry states from sampled inputs.
  - Compares empirical probabilities against the theoretical ripple-carry model.
  - Computes activity factors from sample-to-sample signal toggles.
  - Writes CSV summaries and SVG plots to `analysis_outputs/`.

## Generated files

- `test_vectors.mem`
  - Random stimulus file generated by `test_vector_gen.py`.
  - Read by `tb.sv`.

- `adder16_tb.out`
  - Compiled simulation executable produced by `iverilog`.

- `adder16_random.vcd`
  - VCD waveform dump from the testbench.
  - Parsed by `model_scripy.py`.

- `simulation_results.csv`
  - Per-sample simulation results written by the testbench.
  - Includes input values, DUT outputs, expected output, and a match flag.

- `analysis_outputs/`
  - Folder containing the analysis CSV files and SVG plots.
  - Includes probability comparisons, absolute-error plots, histograms, and activity-factor plots.

## What the probabilistic model does

For a single full-adder stage with independent unbiased inputs:

- `P(C_out = 1 | C_in = 0) = 1/4`
- `P(C_out = 1 | C_in = 1) = 3/4`
- `P(S_i = 1 | C_in = 0) = 1/2`
- `P(S_i = 1 | C_in = 1) = 1/2`

This gives the carry recursion:

`p_{i+1} = 1/4 + (1/2)p_i`

When `P(C_in = 1) = 0.5`, the carry process is already at steady state, so:

- each carry bit is expected to be `1` with probability `0.5`,
- each sum bit is expected to be `1` with probability `0.5`,
- and activity factors are expected to be close to `0.5` under independent random inputs.

The analysis script checks whether the empirical simulation data agrees with these predictions.

You need:

- `python3`
- `numpy`
- `iverilog`
- `vvp`

Check that the simulator tools are installed:

```bash
which iverilog
which vvp
python3 --version
```

Install `numpy` if needed:

```bash
python3 -m pip install numpy
```

## How to reproduce the results

Run the following commands from the repository root.

### 1. Generate the random test vectors

```bash
python3 test_vector_gen.py
```

This creates:

- `test_vectors.mem`

By default, the script uses:

- `10000` samples
- random seed `24`

You can override these options:

```bash
python3 test_vector_gen.py --num-samples 10000 --seed 24 --output test_vectors.mem
```

### 2. Compile the SystemVerilog files

```bash
iverilog -g2012 -o adder16_tb.out 16bit_fa.sv tb.sv
```

This creates:

- `adder16_tb.out`

### 3. Run the simulation

```bash
vvp adder16_tb.out
```

This creates:

- `adder16_random.vcd`
- `simulation_results.csv`

If the flow is working correctly, the testbench should report:

- `Loaded samples: 10000`
- `Mismatches: 0`

### 4. Run the probabilistic analysis

```bash
python3 model_scripy.py
```

This:

- prints the probabilistic model summary,
- parses `adder16_random.vcd`,
- compares empirical and theoretical results,
- computes activity factors,
- and writes plots and tables into `analysis_outputs/`.

You can also specify a custom VCD path or output folder:

```bash
python3 model_scripy.py --vcd adder16_random.vcd --output-dir analysis_outputs
```

## Expected console behavior

After a successful run, the analysis script should report:

- `Recovered samples: 10000`
- `Functional mismatches between DUT output and software model: 0`
- `Mismatches between VCD expected_total and software model: 0`

It should also conclude that:

- the sum-bit probabilities match theory,
- the carry probabilities match theory,
- the conditional carry transition probabilities match theory,
- the activity factors are close to `0.5`,
- and the VCD data is consistent with the ripple-carry probabilistic model.

## Interpreting the results

- If `Mismatches: 0`, the hardware simulation is functionally correct for the sampled inputs.
- If empirical probabilities stay close to the model values, the adder’s statistical behavior matches the ripple-carry model.
- If activity factors are close to `0.5`, the sampled random input assumption is behaving as expected.
- Small absolute errors indicate finite-sample variation rather than a design flaw.

In other words, the project validates both:

- logical correctness,
- and statistical correctness.

## Notes on real-world interpretation

This repository uses independent random inputs as a baseline model. Real processor and accelerator workloads are often not fully random. In practice:

- addresses,
- counters,
- branch-related values,
- memory traffic,
- and compiler-optimized program data

can create correlated and partially predictable inputs.

That means real activity factors are often lower than `0.5`. So this model is best treated as:

- a clean theoretical baseline,
- not a final power model for real silicon workloads.

The same framework could be extended by replacing the random-input assumption with measured application traces.
